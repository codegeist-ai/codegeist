# T004 Implement Codegeist Version Flag

Status: solved

## Goal

Implement the first tiny Codegeist CLI behavior: the Spring Shell `--version`
command prints the current application version.

This is the smallest useful implementation step after removing the
placeholder-first plans. It creates real product behavior without introducing an
agent loop, provider integration, tool registration, or future-facing
architecture placeholders.

## Scope

- Work in the existing Maven module under `app/codegeist/cli`.
- Add support for `--version`.
- Print only the version string, for example `0.1.0-SNAPSHOT`.
- Source the version from the existing project/build or Spring configuration where
  practical so it is not duplicated in multiple code locations.
- Keep the command body side-effect free beyond printing the version.
- Add or update focused tests for the version behavior.
- Keep ordinary Spring context tests passing.
- Keep logs out of stdout so command output remains machine-readable.
- Verify the native executable can run the version command.

## Non-Goals

- Do not add Ollama, Spring AI provider calls, model configuration, or
  Testcontainers.
- Do not add an interactive agent loop.
- Do not register Spring AI Agent Utils tools.
- Do not add prompt handling, chat memory, system prompts, sessions, events, or
  workspace behavior.
- Do not create placeholder classes, ids, ports, records, enums, package layers,
  validation hierarchies, or empty package directories.
- Do not add release automation or CI workflows.
- Do not recreate broad implementation handoff documents under
  `docs/developer/implementation/`.

## Acceptance Criteria

- The `--version` command prints the Codegeist version.
- The version output is deterministic and easy to assert in tests.
- A focused test covers the version behavior.
- The existing Spring context test still passes.
- A Spring-context test covers the `--version` command path.
- Process console output contains only the version string while logs go to a
  writable log file.
- `task native-smoke` verifies the built native executable's version output and
  file logging.
- `docs/developer/architecture/architecture.md` is updated if source,
  configuration, or test behavior changes.

## Verification

Planned checks from `app/codegeist/cli`:

```bash
mvn --batch-mode --no-transfer-progress -Dtest=CodegeistApplicationTests test
mvn --batch-mode --no-transfer-progress test
```

If a more focused test class or method is added for `--version`, run that selector
first and then the broader test command.

Actual checks run from `app/codegeist/cli`:

```bash
mvn --batch-mode --no-transfer-progress -Dtest=VersionCommandsTests test
mvn --batch-mode --no-transfer-progress test
mvn --batch-mode --no-transfer-progress -DskipTests package
mkdir -p target/version-check
java -jar target/codegeist.jar --version > target/version-check/console 2>&1
task native-smoke
```

The focused version test and full Maven suite each completed in about two seconds
after dependencies were warm. The native smoke, including native compilation,
completed in about 44 seconds. The packaged-jar process check compared stdout
and stderr combined with the generated build version and verified file logging
was non-empty.

## Solve Notes

- Implemented `--version` as a Spring Shell command in `VersionCommands`.
- Reused `VersionCommands.VERSION_COMMAND` in tests.
- Sourced the runtime version from Spring Boot's `BuildProperties` bean, backed by
  `META-INF/build-info.properties` generated by the Maven `build-info` goal.
- Added `VersionCommandsTests` as a Spring integration test that proves the
  command runs through the Spring Boot context while keeping application logs out
  of stdout and avoiding a hard-coded Maven project version.
- Set `spring.shell.interactive.enabled=false` in `application.yaml` so Spring
  Shell's noninteractive runner handles `--version` by default without adding a
  custom runner class.
- Added GraalVM `resource-config.json` metadata for `logback.xml`, Logback
  version resources, and `META-INF/build-info.properties` instead of Java runtime
  hints.
- Added standalone `logback.xml` configuration that writes logs only to
  `${LOG_FILE:-logs/codegeist.log}` so command console output remains clean.
- Updated `task native-smoke` to compare the native command's combined stdout and
  stderr with the generated build version and require the native smoke log file
  at `target/smoke-test/codegeist.log` to be written.
- Moved the native smoke assertions into `scripts/native-smoke.sh` as the single
  `run-native-smoke-tests` function; `task native-smoke` now builds native first,
  sources that script, and calls the function. The function deletes and recreates
  `target/smoke-test` for each run.
